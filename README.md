# Инструкция по сборке и использованию модуля анализатора памяти

Проект разработан на C++ для операционной системы Windows (целевая архитектура: **x86**) с использованием Visual Studio 2019+.

## Сборка проекта с помощью CMake

Вы можете собрать проект с помощью CMake в командной строке или открыть папку проекта в Visual Studio (которая имеет встроенную поддержку CMake).

### Сборка через командную строку (Visual Studio Developer Command Prompt):

1. Откройте **Developer Command Prompt для Visual Studio** (обязательно выберите x86 Native Tools Command Prompt, так как целевая архитектура — x86).
2. Перейдите в корневую директорию проекта:
   ```bash
   cd /path/to/memory_analyze
   ```
3. Создайте директорию для сборки и перейдите в нее:
   ```bash
   mkdir build
   cd build
   ```
4. Сгенерируйте файлы сборки для x86:
   ```bash
   cmake -G "Visual Studio 16 2019" -A Win32 ..
   # или для VS 2022:
   # cmake -G "Visual Studio 17 2022" -A Win32 ..
   ```
5. Соберите проект:
   ```bash
   cmake --build . --config Release
   ```

После успешной сборки исполняемый файл `TestApp.exe` и библиотека `MemoryAnalyzer.dll` будут находиться в папке `build/Release/`. Библиотека автоматически скопируется в папку к исполняемому файлу.

### Smoke-test

Для автоматической проверки без ожидания ввода можно запустить:

```bash
ctest -C Release --output-on-failure
```

Или напрямую:

```bash
TestApp.exe --no-wait
```

### Релизы

GitHub Actions автоматически собирает и публикует Windows x86 архив при отправке
тега, начинающегося с `v`:

```bash
git tag v1.0.0
git push origin v1.0.0
```

В GitHub Release появятся ZIP-архив с `MemoryAnalyzer.dll`, `Injector.exe`,
`TestApp.exe` и файл с контрольной суммой SHA-256.

### Запуск инжектора

```bash
Injector.exe --pid <process_id> <absolute_path_to_MemoryAnalyzer.dll>
Injector.exe --name <process_name.exe> <absolute_path_to_MemoryAnalyzer.dll>
```

Инжектор создаёт именованный канал `\\.\pipe\MemoryAnalyzerPipe` и печатает логи, которые отправляет DLL.

Перед ручным вызовом `FreeLibrary` необходимо вызвать экспортированную функцию
`MemoryAnalyzerShutdown`, чтобы снять хуки и остановить фоновые потоки вне loader lock.

## Принцип работы

1. **Инъекция:** При загрузке DLL в целевой процесс запускается фоновый поток `MainThread`.
2. **Инициализация хуков:** MinHook устанавливает trampoline-перехватчики на функции выделения памяти и изменения прав доступа.
3. **Первичное сканирование:** Сканер обходит всю виртуальную память процесса с помощью `VirtualQuery` для поиска уже загруженных скрытых PE-модулей.
4. **Мониторинг:** При вызове `VirtualAlloc` или `VirtualProtect` с исполняемыми флагами (например, `PAGE_EXECUTE_READWRITE`), адрес региона отправляется в потокобезопасную очередь сканера.
5. **Дамп и восстановление:** Фоновый поток извлекает адреса из очереди, проверяет наличие сигнатур `MZ` и `PE`, восстанавливает выравнивание секций (преобразует `SectionAlignment` обратно в `FileAlignment`) и сохраняет корректный PE-файл на диск.

## Выходные файлы

* `memory_analyzer.log` — основной лог DLL.
* `test_app.log` — лог тестового приложения.
* `dump_<base_address>_<timestamp>.exe` — восстановленный PE-дамп.

## Ограничения и безопасность

* Проект рассчитан на Windows x86/PE32. Для Visual Studio generator используйте `-A Win32`.
* Для inline-хуков используется MinHook `v1.3.4`, который CMake загружает при первой конфигурации проекта. Для этого машине сборки нужен доступ к GitHub.
* Инструмент предназначен только для анализа собственных процессов или процессов, на которые у вас есть явное разрешение.

## Troubleshooting

* Если `MemoryAnalyzer.dll` не загружается, проверьте, что `TestApp.exe`/целевой процесс и DLL собраны в одной архитектуре x86.
* Если дамп не появляется, посмотрите `memory_analyzer.log`: наиболее частые причины — регион памяти защищён `PAGE_GUARD`, PE-заголовки повреждены или модуль записан после первого сканирования без последующего `VirtualProtect`.
* Если `Injector.exe` не подключает DLL, запускайте его с достаточными правами и передавайте абсолютный путь к DLL.
